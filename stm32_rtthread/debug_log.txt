调试日志：
一.oled显示异常
1. 现象：当缓冲数据峰值大于环形缓冲区容量时OLED 屏幕显示少量杂乱像素，疑似显示数据被污染。
2. 逻辑分析仪：观察到 clear 命令发送时并非全 0，有少量数据残留，说明传输中已有异常。
3. 调试：在环形缓冲区写入函数打断点，记录写入行为，查看内存十六进制数据。
4. 定位：当写入数据帧位于缓冲区尾部时，仅用 memcpy 直接拷贝，没有分段处理；写指针回环后下一帧还是从头开始，导致越界写入尾部剩余空间，造成当前帧数据异常。
5. 修复：在写函数里判断是否跨越尾部，需要做分两段写,读取函数也不能采用零拷贝的方式,要改成先写入
缓冲区以解决回环时i2c传输函数只能线性传输的问题
修复完成后发现还是未能解决问题,在write函数打断点,观察rb内存写入情况,发现能够正常回环,并且写入的数据帧内
数据部分都是0,又通过逻辑分析仪观察波形,发现oled收到的数据并不全是0,疑似read函数有问题,经过排查发现
找不到read函数的问题,重新观察写入内存布局发现每次数据段写入0的并不是完整一行,对write函数重新排查后
发现是因为frame_head结构体指针没有先转为uint8*在加减再转void*,导致memcpy写入的偏移地址算错,数据
错位.
再次修复后发现oled每行末段还是会出现随机像素,且每次复位时位置不同,可能是末段写入时临时读缓冲区
数据失效了,排查发现是因为把它设为局部变量过早释放了,改为rt_malloc获取,中断内free问题解决



二.oled数据调小间隔调小后rb卡死
1.现象:程序运行一段时间后oled上的数据完全不动,逻辑分析仪抓取不到任何波形
2.调试:oled卡死后暂停程序发现rb_write的返回值一直是-1导致cpu一直忙等,表明rb一直是满的,中断发送服务函数没有
正常调用,对rt_read函数加上错误调试信息后得到RB Read Error! FrameLen:1536, BufSize:129,表明是读取到了错误
的数据帧头,给数据帧头添加magic成员固定为6,打印错误帧头的magic,结果为3,表明很可能是read指针错位了,又在每次
read函数后面加上打印相关调试信息,得到的错误信息及部分上文:
Len:2 read:1428 write:1364
Len:2 read:1432 write:1364
rbLen:2 read:1436 write:1430
:2 Len:19 read:1457 write:1430
read:1457 Len:31 read:1490 write:1442
Len:2 read:1494 write:1475
Len:2 read:1498 write:1475
Len:2 read:2 write:1475
Len:1536 read:2 write:8
RB Read Error! FrameLen:1536, BufSize:129
可以发现read指针并没有错位,而是write函数在空间不足的情况下异常写入数据,导致覆盖了read指针指向的数据,
正常来说这是不可能出现的,对相关函数一一分析后发现是因为判断剩余空间的函数出现了竞态:
uint16_t rb_available_space(rb_t *rb)
{
    if (w >= rb->read) return rb->max_len - (rb->write - rb->read) - 1;
    else return rb->read - rb->write - 1;
}
在上面的情景中,rb->write 为 1475，rb->read 为 1498。
if (rb->write >= rb->read) 判断为 假。
此时发生中断，rb_read 被调用，rb->read 更新为 2。
执行 else 分支：return rb->read - rb->write - 1。
此时读取到的 rb->read 是 2，计算结果为 2 - 1475 - 1 = -1474。
由于返回类型是 uint16_t，-1474 会溢出变成一个巨大的正数（ 64062）。
rb_write 看到剩余空间是 64062，认为空间充足，于是执行了写入。
将函数修正为:
uint16_t rb_available_space(rb_t *rb)
{
    uint16_t w = rb->write; 
    uint16_t r = rb->read;  
    if (w >= r) return rb->max_len - (w - r) - 1;
    else return r - w - 1;
}
如果中断发生在两行之间：拿到了最新的 r。计算出的剩余空间会比中断前更大。这是安全的，因为空间确实变多了。
如果中断发生在两行之后：拿到了旧的 r。计算出的剩余空间会比实际更小。这也是安全的，因为这只会保守地认为空间不足，不会导致溢出。
这样既避免了不必要的线程阻塞导致的上下文切换,又解决了函数的竞态问题.
